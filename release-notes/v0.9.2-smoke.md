## Findings

### Finding A — `receipt_signing_key_id` was missing from `TenantConfigPatch`

**Status:** real gap. **Resolved in [statewave#173](https://github.com/smaramwbc/statewave/pull/173) during this v0.9.3 patch.**

**What I observed.** I PATCHed `/admin/tenants/{id}/config` with `{"receipt_signing_key_id": "smoke-key-1"}`. The request returned **200** but the response showed `config: {}` — the field had been silently dropped. The receipt emitted under that tenant came out unsigned (`valid: null, reason: "no_signature"`).

**Root cause.** `TenantConfigPatch` (in `server/schemas/requests.py`) didn't list `receipt_signing_key_id`. Pydantic's default `extra="ignore"` discarded the unknown field. The HMAC signing infrastructure (`services/receipts.py`) was reading from `tenant_configs.config.receipt_signing_key_id`, but operators had **no API surface to set it** — they had to write the JSONB directly via psql.

**Workaround used during initial smoke.** Direct SQL `UPDATE tenant_configs SET config = jsonb_set(...)`. Worked, but unacceptable for normal operator workflow.

**Fix.** [statewave#173](https://github.com/smaramwbc/statewave/pull/173) added the field (with shallow length+charset validation, no key-bytes lookup), plus `extra="forbid"` on the schema so the silent-drop class of bug surfaces as 422 in the future.

**Post-fix verification (this report's re-run).** PATCH-only HMAC config now works end-to-end:

```
PATCH /admin/tenants/hmac-via-patch/config  {"receipt_signing_key_id":"smoke-key-1"}  → 200
config persisted: {"receipt_signing_key_id":"smoke-key-1"}
emit_receipt → verify → valid=True, key_id=smoke-key-1, algorithm=hmac-sha256-canonical-v1
```

### Finding B — region PATCH (initially flagged, **confirmed false alarm**)

**Status:** not a real finding. Initial smoke flagged this on assumption; re-check disproved it.

**What I initially observed.** I assumed region had the same gap as A because I went straight to SQL for the residency portion of the smoke without first trying PATCH.

**On re-check.** `region` was added to `TenantConfigPatch` during the v0.9 [#161](https://github.com/smaramwbc/statewave/pull/167) residency PR — present on line 230 of `server/schemas/requests.py`, with the full `validate_region_pin` safety check and `force_region_pin` bypass. **Four integration tests in `tests/integration/test_residency.py`** cover the PATCH path:
- `test_admin_patch_accepts_pin_to_local_region`
- `test_admin_patch_refuses_pin_to_foreign_region_without_force`
- `test_admin_patch_accepts_force_region_pin`
- `test_admin_patch_allows_any_pin_in_single_region_mode`

All four PASS on current main.

**Post-fix verification (this report's re-run).** PATCH-only region pin works end-to-end:

```
PATCH /admin/tenants/region-via-patch/config  {"region":"us", "force_region_pin":true}  → 200
config persisted: {"region":"us"}
request as that tenant  → 403  error.code=residency.mismatch
```

**Lesson.** I should have tested PATCH before assuming the gap. Caught and corrected before any code was written for the false alarm; no false-Finding-B PR was opened.

### Finding C — `session_id` not on the Python SDK's `create_episode`

**Status:** known gap, **out of scope for v0.9.3**.

**What I observed.** During the handoff replay smoke, calling `sw.create_episode(..., session_id="ho-sess")` raised `TypeError: unexpected keyword argument 'session_id'`. I worked around with raw `httpx`.

**Pre-existing acknowledgment.** Already documented in `statewave-examples/eval-support-agent/README.md:30` ("session_id on episode creation, on get_context, and the /v1/handoff and /v1/resolutions endpoints aren't on the SDK yet"). Not a new finding from this smoke.

**Tracked as:** an SDK follow-up — open a tracking issue if not already on the SDK roadmap. Target: a future Python/TypeScript SDK patch release. **Not blocking v0.9.3 or v1.0** since the examples already work around it.

### Finding D — Docker image cache lag for returning users

**Status:** cosmetic, **addressed in v0.9.3 onboarding polish**.

**What I observed.** My local Docker had a 10-day-old cached `:latest` image. Docker Hub had a current `:latest` published 23:17 UTC. `docker compose up -d` on a stale cache doesn't auto-update — the host's cached image wins until `docker compose pull` runs explicitly.

**Impact.** A brand-new user (no cache) is unaffected — `up -d` pulls fresh. A returning user (cached `:latest` from a previous run) may stay on a stale version without realizing it.

**Fix.** Adding a single returning-user callout to the onboarding polish PR in v0.9.3:

> If you have run Statewave before, run `docker compose pull` before `docker compose up -d` to refresh cached images.

No Docker tagging project, no image-build changes.

## What the smoke confirms

End-to-end through the documented operator workflow, on v0.9.3 main:

- A new operator can `git clone` + `docker compose pull` + `docker compose up -d` and have a working Statewave server in under two minutes.
- The current published SDKs (`statewave==0.10.1`, `@statewavedev/sdk@0.10.1`) install cleanly and expose the v0.9 governance methods (verify, replay).
- HMAC signing is enable-able **purely via the admin API** — no SQL required.
- Tenant residency is pin-able **purely via the admin API** — no SQL required, with the safety check + force-bypass both working as documented.
- Auto-labeling round-trip works (detector → suggestion → admin review → promote → authoritative `sensitivity_labels`).
- Receipt verify + replay work in both happy and refusal paths, including the v0.9 #168 handoff replay symmetry.

**Confidence for v1.0 launch:** the v0.9 governance + replay + residency stack is solid through the operator workflow. The remaining v0.9.3 work (proof-figure sweep, onboarding polish, examples, optional governance quickstart, connector skim) is polish on top of a verified base.
