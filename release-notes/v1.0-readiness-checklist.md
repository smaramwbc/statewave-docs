## 3. Are limitations honest?

✅ **Yes — the published "Still in progress / not yet proven" section in `why-statewave.md` matches the carry-forward risk list in every release-notes file.**

The honest list (carried into v1.0):

- Production-scale not load-tested at >10k subjects / high throughput. Multi-replica API supported and verified since v0.8 (Fly multi-machine + Helm HPA); **single-Postgres only, no cross-region clustering.**
- Multi-tenant isolation is app-layer query scoping; no Postgres RLS yet.
- LLM-compiler vs heuristic-compiler comparative eval not yet published.
- 50-session production-scale benchmark not yet run.
- No second-region deploy verified (residency enforcement is implemented and audited per-region; cross-region failover is not tested because we run one region today).
- No admin-action identity yet (`promoted_by` is null until that layer ships).
- No federated cross-region audit.
- Receipt replay is "current code + original policy" (byte-for-byte historical reproduction needs memory snapshots, deferred).

## Explicit claim boundary for v1.0

Now that the v1.0 tag is cut (2026-06-09), the workspace surfaces say:

> **v1.0 — first stable public developer release.**

The workspace surfaces MUST NOT say (per the v0.9.4 plan approval):

- GA
- production-ready
- production-hardened
- battle-tested
- enterprise-ready
- hardened

A v1.0 launch user can expect:

1. The documented quickstart works against the published image.
2. Both SDKs install cleanly from their public registries and the typed surface matches the REST contract.
3. The published "Still in progress / not yet proven" list is honest about what is NOT proven yet.
4. Receipts + sensitivity labels + policy + residency are all reachable from the API and the admin console.
5. Multi-replica API deployment is supported (single-Postgres only).

What a v1.0 launch user MUST NOT expect, and what the public docs say so:

1. Production-scale load testing at >10k subjects.
2. Postgres RLS for tenant isolation.
3. A second region you can fail over to.
4. Admin-action identity (`promoted_by`).
5. Federated cross-region audit.
6. Byte-for-byte historical replay.
7. LLM-compiler-vs-heuristic comparative eval results.
