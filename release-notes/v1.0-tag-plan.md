## 2. Final wording (locked)

The maintainer-approved phrasing for v1.0 surfaces. **No GA / production-ready / production-hardened / battle-tested / enterprise-ready / hardened anywhere.**

### Canonical lead line

> **Statewave v1.0 — first stable public developer release.**

### GitHub Release title (`statewave` repo)

> `v1.0.0 — first stable public developer release`

### GitHub Release body lead paragraph

> Statewave v1.0 is the first stable public developer release of the open-source memory runtime that gives AI agents reproducible, provenance-tagged context. The v1 API contract (`/v1/*`) and the v0.9 governance surfaces (HMAC-signed receipts, replay, sensitivity labels with declarative policy, opt-in detector-suggested labels, per-region residency pinning) are now considered stable for developer use under a self-hosted deployment model. Limitations are documented in [`why-statewave.md`](https://github.com/smaramwbc/statewave-docs/blob/main/why-statewave.md); see [`v1.0-readiness-checklist.md`](https://github.com/smaramwbc/statewave-docs/blob/main/release-notes/v1.0-readiness-checklist.md) for the full sign-off artefact.

### What the wording explicitly does NOT promise

The launch body / post drafts MUST NOT include any of these phrases:

- "production-ready", "production-hardened", "production-grade"
- "battle-tested", "hardened", "enterprise-ready", "enterprise-grade"
- "GA", "general availability"
- Any claim of compliance certification (SOC2, ISO, HIPAA, etc.) that Statewave has not undertaken
- Any benchmark figure that is not in the canonical proof-figure table at the foot of `v0.9.4-launch-readiness.md`

### What it DOES say honestly

- The /v1 API contract is stable.
- Multi-replica API supported and verified since v0.8.
- Single-Postgres only; cross-region clustering is roadmap.
- Self-hosted; no managed plane.
- The Statewave IDE Companion is available as a preview on the VS Code Marketplace and Open VSX (Cursor / Windsurf), v0.1.6.
- The "still in progress / not yet proven" list in `why-statewave.md` is authoritative; nothing claimed at launch contradicts it.

## 6. Rollback / supersede policy (per [[feedback-published-tag-immutability]])

**Once a tag is pushed, it is immutable.** Force-moving a tag breaks every consumer that already pinned to it.

### If a v1.0.0 mistake is found *before* the GitHub Release is published

The tag exists but no announcement has shipped. You have a small window:

- **Don't** force-move v1.0.0.
- Cut `v1.0.1` at the corrected commit, push that tag. v1.0.0 becomes a quiet pre-release with no GitHub Release page.
- Publish the GitHub Release **only** at v1.0.1.

### If a v1.0.0 mistake is found *after* the GitHub Release is published

The community is now pinned to it. Per the published-tag-immutability discipline:

- Cut `v1.0.1` at the corrected commit.
- Edit the v1.0.0 GitHub Release page: prepend `**⚠️ SUPERSEDED by [v1.0.1](…)**` and a one-sentence reason. Do NOT delete the release.
- Cut a new GitHub Release at v1.0.1 with title `v1.0.1 — corrected v1.0 release` and a short body explaining the fix.
- Publish a short social note pointing pinned-version users at v1.0.1.

### If Docker Hub `:latest` was wrong

Easier — push a corrected `:latest` on top. But also:

- Re-tag the corrected image as `:1.0.1` so anyone pinning by version sees the same bytes.
- Note in the GitHub Release that `:1.0` floats to `:1.0.1`.

### If a tag was cut on the wrong commit entirely

Same as "mistake found after release" — never force-move. Always supersede. **The user's standing direction is explicit on this point: OSS trust depends on consumers' pins not silently shifting.**
