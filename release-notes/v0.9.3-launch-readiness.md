## Risks carried into v0.9.4 / v1.0

These are the items the workspace has *not* verified, listed honestly so an evaluator does not learn about them at launch.

1. **Production scale not load-tested.** Multi-replica API supported and verified since v0.8 (Fly multi-machine + Helm HPA); single-Postgres only, no cross-region clustering. `why-statewave.md` says the same; no new evidence in v0.9.3. (Wording corrected post-publication via the v0.9.4 launch-doc-freeze audit — the earlier "Single-node only" phrasing here misrepresented the multi-replica API support.)
2. **Multi-tenant isolation is app-layer.** No Postgres RLS. Not battle-tested at scale.
3. **SDK governance ergonomics are incomplete.** [#176](https://github.com/smaramwbc/statewave/issues/176) tracks the gap. Operators who want auto-labeling today need raw HTTP for the read-and-promote story.
4. **The repo-map "independent" version cells will drift again.** They are NOT in the auto-bump path by design (each SDK versions independently). The v0.9.3 sweep just caught up two releases of drift; the next SDK bump will start the drift again unless the release checklist (`tools/RELEASE-CHECKLIST.md`) is followed for each SDK.
5. **No second-region deploy.** Residency enforcement is implemented and audited; cross-region failover is not tested because there is no second region today.

None of these block the 2026-06-16 v1.0 *candidate* — they are part of the published "still in progress / not yet proven" list. They block any claim of "production-hardened" or "battle-tested." Don't make that claim until the corresponding box has been ticked.
