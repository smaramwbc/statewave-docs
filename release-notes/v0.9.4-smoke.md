## Findings

### Finding A — soft — no public version endpoint — **deferred**

`GET /version` returns 404. `GET /v1/version` returns 404. The server has no programmatic version-discovery surface; the only way to learn the running version is `pip show statewave` inside the container, or the Docker image tag.

**Impact**: low. Operators almost always know which tag they pulled. But it's the kind of thing evaluators check during a security review or a multi-version deployment audit, and saying "we don't expose it" is a worse story than shipping a one-line endpoint.

**Disposition**: tracked as [statewave#178](https://github.com/smaramwbc/statewave/issues/178) for v0.10.x. **Not a v1.0 blocker.** Not on the launch demo path; documented workaround exists (Docker tag / pip metadata inside the container). Deferring per the v0.9.4 guardrails — not spending lap capacity here.

### Finding B — medium — subject-scoped memory listing omits label fields — **fixed in-stream (#177)**

**Original observation (pre-fix, SHA `da8cc87`):** `GET /admin/subjects/{id}/memories` returned 10 fields per memory: `confidence, content, created_at, id, kind, source_episode_ids, status, summary, valid_from, valid_to`. Neither `sensitivity_labels` nor `suggested_labels` was present. The dedicated `/admin/memories/with-suggested-labels` route did include them, but that endpoint only shows memories that *currently have un-promoted suggestions* — once an admin promoted labels, the memory disappeared from that view, and there was no admin-flow path to see authoritative labels on the subject's full memory listing.

**Why this mattered:** the natural admin demo flow ("open subject → view memories → see labels") was broken. The most visible v0.9 governance UX gap an evaluator could hit during a 30-minute demo.

**Fix shipped at [statewave#177](https://github.com/smaramwbc/statewave/pull/177), merged at SHA `fe82b9b`, published to `:latest` 2026-05-27T16:33Z.** Two new fields added to `MemoryListItem`. Purely additive. No migration, no breaking move, no behaviour change on any existing field. Two new integration tests in `tests/integration/test_auto_labeling.py` (`test_subject_memory_listing_includes_label_fields_after_suggestion` and `test_subject_memory_listing_reflects_promoted_labels`).

**Re-run validation** (against `:latest` at `fe82b9b`):

```bash
# Pre-fix shape (left for the audit trail):
# → ["confidence","content","created_at","id","kind","source_episode_ids","status","summary","valid_from","valid_to"]

# Post-fix shape (verified 2026-05-27 post-republish):
$ curl -s :8101/admin/subjects/smoke-pii-v2/memories | jq '.memories[0] | keys'
# → ["confidence","content","created_at","id","kind","sensitivity_labels","source_episode_ids","status","suggested_labels","summary","valid_from","valid_to"]
$ curl -s :8101/admin/subjects/smoke-pii-v2/memories | jq '.memories[0] | {suggested_labels, sensitivity_labels}'
# After compile (PII payload — email + Luhn-valid card):
# → {"suggested_labels": ["financial.card","pii.email"], "sensitivity_labels": []}

# After POST /admin/memories/{id}/promote-labels {"labels": ["pii.email","financial.card"]}:
# → {"suggested_labels": [], "sensitivity_labels": ["financial.card","pii.email"]}
```

Both arrays are present, populated correctly with the heuristic detector hits, and reflect the suggested → authoritative move atomically in the listing. Scenarios 6 and 9 re-ran cleanly against the post-fix image.

## Positive evidence worth recording

These were not asked-for findings but are worth tracking because they materially shape the v1.0 story:

- **statewave#173 is in `:latest`.** PATCH `receipt_signing_key_id` works end-to-end against the published image; `extra="forbid"` rejects unknown fields with a clear validation error. The v0.9.3 in-stream fix shipped intact.
- **Residency foot-gun guard.** The PATCH route refuses to set a tenant pin to a region the current server doesn't serve, with a helpful remediation hint. This is the kind of "we sweated the operator UX" detail worth highlighting in the launch post.
- **Residency 403 does not echo the server's actual region.** Audit boundary intact. The 403 body names the tenant's pinned region (so the caller knows where to retry) but never names the server's region.
- **Receipt replay is byte-identical.** The replay receipt's `diff.context_hash.changed` is `false` for both `verify+replay` and admin-flow replay. Replay determinism is what the receipt story is for; it works.
- **Auto-labeling heuristics fire correctly.** `pii.email` matched on `alice@globex.com`, `pii.phone` on `555-867-5309`, `financial.card` on a Luhn-valid `4242 4242 4242 4242`. All three came back in `suggested_labels` and none in `sensitivity_labels` (correctly — they need explicit promotion).
- **Server log scan was clean.** No `error`, `exception`, `critical`, or `fail` lines during the full smoke (`docker logs sw_smoke_094-api-1` after the run). The only warnings were the expected `STATEWAVE_LITELLM_API_KEY is not set` (demo-mode running).

## What was NOT exercised in this smoke

Intentional, per the v0.9.4 guardrails:

- LLM compiler path (heuristic-only is the demo-mode path)
- LiteLLM embeddings (stub embeddings only)
- Multi-tenant load (one tenant: `default`)
- Auth (running in `authDisabled=true` mode — admin console reports `source: disabled`)
- Webhook delivery
- Signed-receipt HMAC verify (requires `STATEWAVE_RECEIPT_SIGNING_KEYS` plus a key id; the PATCH surface that wires this together is exercised, but emitting a signed receipt + verifying its HMAC requires real key material — out of smoke scope, covered by server unit tests instead)
- Cross-region failover (single region, single node)

These match the published "Still in progress / not yet proven" section in `why-statewave.md` and the v0.9.3 close-out's "carried into v0.9.4 / v1.0" list.
