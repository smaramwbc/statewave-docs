## Risks carried into v1.0

These are the items the workspace has *not* verified, listed honestly so an evaluator does not learn about them at launch. Identical to the v0.9.3 list except the wording is now precise (the v0.9.4 audit Finding 1 fix).

1. **Production-scale not load-tested** at >10k subjects / high throughput. Multi-replica API supported and verified since v0.8 (Fly multi-machine + Helm HPA); **single-Postgres only, no cross-region clustering.**
2. **Multi-tenant isolation is app-layer.** No Postgres RLS.
3. **SDK governance ergonomics incomplete.** Tracked at [statewave#176](https://github.com/smaramwbc/statewave/issues/176). Auto-labeling is opt-in; the default v1.0 user does not hit this.
4. **The repo-map "independent" version cells will drift again.** They are NOT in the auto-bump path by design. v0.9.4 caught up the v0.10.2 wave; following [`tools/RELEASE-CHECKLIST.md`](../tools/RELEASE-CHECKLIST.md) on every SDK release keeps it honest.
5. **No second-region deploy verified.** Residency enforcement is implemented and audited per-region; cross-region failover is not tested because we run one region today.
6. **LLM-compiler vs heuristic-compiler comparative eval** not yet published.
7. **50-session production-scale benchmark** not yet run.

None of these block the 2026-06-16 v1.0 *candidate* — they are the published "still in progress / not yet proven" list. They block any claim of "production-hardened" or "battle-tested." Don't make that claim until the corresponding box has been ticked.
